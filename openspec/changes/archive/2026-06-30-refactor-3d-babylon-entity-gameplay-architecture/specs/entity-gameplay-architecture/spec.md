## 新增需求

### 需求：架构分类必须明确
3D 模块必须将运行时代码组织到明确的架构分类中，而不是使用宽泛的 `systems/` 桶同时承载玩法和基础设施。

#### 场景：新增运行时服务
- **当** 代码提供输入、音频、路由、实体生命周期、交互检测、配置或宿主外壳桥接等可复用基础设施时
- **则** 它必须放在面向服务的模块边界中，例如 `services/` 或 `core/`，而不是放在玩法专属模块中

#### 场景：新增玩法领域
- **当** 代码实现农场、PK、建筑、家园编辑、街机、扑克、任务、商店、背包或排行榜行为等领域规则时
- **则** 它必须放在面向玩法的模块边界中，例如 `gameplay/<domain>/`

### 需求：实体代表可交互世界对象
3D 模块必须将可交互世界对象表示为拥有清晰生命周期所有权的实体。

#### 场景：实体拥有 Babylon 节点
- **当** 一个世界对象创建或包装 Babylon 节点、碰撞体、交互体、阴影投射体或更新行为时
- **则** 拥有该对象的实体必须暴露更新和释放所需的生命周期行为，而不要求 `main.js` 直接管理内部 mesh

#### 场景：选择实体命名
- **当** 引入新的实体类型时
- **则** 其名称必须简单且领域含义明确，例如 `Player`、`NPC`、`Building`、`FarmPlot`、`Crop`、`Weapon`、`Bomb`、`Portal` 或 `BeachBall`

### 需求：组件表达可复用能力
3D 模块必须通过组件式组合支持可复用实体能力，但不要求完整 ECS 实现。

#### 场景：需要可复用交互能力
- **当** 多种实体类型需要共享能力，例如可渲染、可碰撞、可交互、可投射阴影、生命值、可种植、可拾取、背包或可受伤行为时
- **则** 该能力必须抽取为可复用的组件式模块，而不是在每个玩法领域中重复实现

### 需求：生成器创建地图内容和实体
3D 模块必须使用生成器从布局、主题配置、预设或蓝图数据构建地图内容和实体化 bundle。

#### 场景：加载地图
- **当** 加载大厅、小屋、农场、PVP、湖或城堡等地图时
- **则** 对应生成器必须通过清晰的地图 bundle 或 world manager 合同返回或注册已创建实体、碰撞体、交互体、阴影投射体、出生点数据和可选更新钩子

### 需求：主入口只负责组合
3D 模块必须让主入口专注于启动和组合，而不是承载领域特定玩法逻辑。

#### 场景：主应用启动
- **当** 3D 模块从 `main.js` 初始化时
- **则** `main.js` 必须将引擎设置、世界加载、服务、实体、玩法模块、UI 面板和宿主集成委托给专门模块
