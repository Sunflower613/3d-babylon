# host-config-boundary Specification

## Purpose
定义 3D 模块如何接收配置、外壳、SSO、音频与 toast 等父站/平台关注点，并将游戏内容数据集中于内部 data 层，使模块可移植、边界清晰，而不是硬编码父目录导入或运行时全局注入。

## 需求

### 需求：游戏内容数据集中于内部 data 层
3D 模块必须将主题、商品、种子、任务与武器等游戏内容数据集中在模块内部的 data 层（`src/data/*`），作为构建期数据源，而不是散落在各玩法/服务模块中硬编码，也不依赖运行时全局注入。

#### 场景：玩法读取内容数据
- **当** 玩法逻辑需要种子成熟时间与收割奖励、武器伤害与击退、初始任务或初始背包等内容数据时
- **则** 它必须从 data 层模块导入这些数据，而不是在玩法代码中硬编码字面量

#### 场景：配置变更
- **当** 需要调整主题颜色、商品列表或玩法数值时
- **则** 修改 data 层数据并重新部署即可生效，模块不得依赖 `window.__SITE_CONFIG__` 等运行时全局注入实现“热更新”

### 需求：配置 provider 优先 data 层并保留 fallback
3D 模块必须通过配置 provider 消费配置数据，provider 优先使用内部 data 层，并保留 `site-config.js` 作为兼容回退，而不是从父目录直接导入作为核心合同，也不再支持运行时全局注入。

#### 场景：解析主题与商品
- **当** 3D 模块需要激活主题配置或商店商品数据时
- **则** 配置 provider 必须优先返回 data 层数据；当 data 层缺失对应项时回退到 `site-config.js`

#### 场景：独立开发运行
- **当** 3D 模块在没有父站宿主的情况下运行时
- **则** 它必须支持足以渲染和测试模块的开发 fallback 配置（data 层 + `site-config.js`）

### 需求：宿主外壳行为通过桥接访问
3D 模块必须通过显式 shell bridge 访问宿主外壳行为，而不是直接散布 `window.parent` 和父级 document 假设。

#### 场景：请求侧边栏或父级导航
- **当** 3D 模块需要打开侧边栏、导航到父级路由、返回 2D 站点或协调 iframe 行为时
- **则** 它必须调用 shell bridge 抽象，该抽象可由父级宿主或独立 fallback 实现

### 需求：SSO 和用户身份归宿主所有
除非显式使用独立 fallback，3D 模块必须将 SSO token 处理和用户身份视为宿主提供的平台数据。

#### 场景：3D UI 需要用户状态
- **当** 3D 模块需要当前用户、登录状态、登出行为或登录跳转行为时
- **则** 它必须通过宿主/平台适配器访问这些操作，而不是将父站 SSO 逻辑嵌入核心运行时模块

### 需求：音频和 toast 服务通过适配器共享
3D 模块必须通过服务处理音效和 toast 消息等跨切面 UI 反馈，这些服务可以委托给宿主，也可以本地运行。

#### 场景：玩法播放音效
- **当** 玩法需要播放短音效时
- **则** 它必须使用音频服务；该服务在宿主音频函数存在时委托给宿主，否则独立运行时回退到本地 Web Audio

#### 场景：玩法显示 toast
- **当** 玩法需要显示状态消息时
- **则** 它必须使用 toast 服务，而不是在玩法逻辑中手动创建 DOM toast 节点
